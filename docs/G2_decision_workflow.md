## 1. OVERVIEW

PulseGuard is a five-phase pipeline. The phases execute sequentially by gate but share a unified evidence ledger.

```
Phase 1 — PRE-TRAINING      (G3)
Phase 2 — TRAINING          (G4, G6)
Phase 3 — DECISION          (G7)
Phase 4 — MONITORING        (G4, G5, G7)
Phase 5 — GOVERNANCE        (G8)
```

Each stage has a defined input contract, an output contract, and a gate condition. A gate condition defines what PASS/WARN/FAIL means and what happens next.

## 2. PHASE 1 — PRE-TRAINING PIPELINE

**Execution context:** Batch, once before any model training. Must complete with PASS or documented WARN before G4 training begins.

### Stage 1 — Raw Feature Input

```
Source: Home Credit Default Risk dataset (public, Kaggle)
        OR synthetic fallback (if Home Credit unavailable — see G1_dataset_plan.md)

Input:  application_train.csv or synthetic_applicants.csv
Format: tabular, ~30–122 columns, one row per applicant, TARGET column present
Key:    SK_ID_CURR (unique applicant ID)

Note:   No feature-level timestamps present natively in Home Credit.
        Synthetic timestamps added at G3 by scripts/add_synthetic_timestamps.py
        to enable FLL temporal leakage checks (Checks 6 & 7).
```

### Stage 2 — Feature Engineering / ABT Construction

```
Action: Assemble Application Base Table (ABT) from raw features.
        Select the ~30 features used for modeling (from 122-column full schema).
        Impute missing values (median for numeric, mode for categorical).
        Encode categoricals (one-hot or target encoding — defined at G4).
        Stratified 60/20/20 train/val/test split on TARGET.

Output: X_train, X_val, X_test, y_train, y_val, y_test (numpy arrays or DataFrames)
Source: scripts/feature_pipeline.py (DEFERRED — G4)

SYNTHETIC note: If Home Credit is unavailable, the ABT is generated by
                scripts/generate_synthetic_data.py (DEFERRED — G3).
                This must be labeled SYNTHETIC in all downstream artifacts.
```

### Stage 3 — Leakage Gate

```
Action: Run FeatureLeakageLens 7-check audit on X_train, y_train.
        Checks: name heuristic, ID proxy, target correlation, categorical proxy,
                split distribution, temporal availability, training future date scan.
        Temporal checks require synthetic timestamps (added at Stage 2b).

Gate conditions:
  FAIL  → Block training. Do not proceed to Phase 2. Escalate to data team.
           Log FAIL findings in outputs/evidence/leakage_report.json.
  WARN  → Document all WARNs. Proceed with explicit caveat in model card.
           Each WARN must be explained (e.g., INSUFFICIENT_INPUT for split check).
  PASS  → Continue. All 7 checks clear at PASS level.

Output: outputs/evidence/leakage_report.json (schema in G2_data_contracts.md)
Source: scripts/leakage_audit.py (DEFERRED — G3)

Expected result on Home Credit:
  Checks 1–5: PASS or WARN (no native temporal leakage in Home Credit)
  Checks 6–7: WARN or INSUFFICIENT_INPUT without synthetic timestamps;
              FAIL on injected leakage column with synthetic timestamps
  SR-12 (source-reference): 3 WARNs, 0 FAILs — not PulseGuard-built yet
```

## 3. PHASE 2 — TRAINING PIPELINE

**Execution context:** Batch, after Phase 1 PASS. Produces champion and challenger models and their evaluation artifacts.

### Stage 4 — Champion Training

```
Action: Train XGBoost on X_train (binary classification, 8% default rate).
        Optimize via Optuna (50 trials, TPE, maximize PR AUC on X_val).
        Select best trial as base champion (xgb_v1 or best_trial).
        Apply Platt sigmoid calibration: CalibratedClassifierCV on X_val, cv='prefit'.
        Evaluate: ROC AUC, PR AUC, ECE on X_test.

Primary metric: PR AUC (class imbalance; PR AUC is more informative than ROC AUC at 8% base rate)
Secondary metric: ECE (calibration gate — ECE regression blocks deployment even if AUC improves)
ECE target: < 0.01

Output: outputs/evidence/calibration_report.json
        outputs/models/champion_xgb.pkl (not a claim artifact — a build artifact)
Source: scripts/train_champion.py, scripts/optuna_hpo.py (DEFERRED — G4)

ECE regression case (from source-reference SR-4/SR-5):
  Optuna may find a model (xgb_v2) with higher PR AUC but worse ECE.
  That model is HELD — not deployed. This demonstrates the calibration gate.
  xgb_v1 is champion. xgb_v2 is documented as "held for ECE regression."
```

### Stage 5 — Champion Evaluation

```
Action: Compute full evaluation suite on X_test for champion model:
  - ROC AUC, PR AUC
  - ECE (global)
  - Calibration curve (predicted vs. observed probability; plotted at decision boundary points)
  - Confusion matrix at each policy threshold (0.06, 0.28)
  - Score decile table (bad rate by decile)
  - SHAP TreeExplainer: mean |SHAP| rank per feature (top 15)

Output: outputs/evidence/calibration_report.json
        outputs/plots/calibration_curve.png
        outputs/plots/shap_beeswarm.png
Source: scripts/train_champion.py, scripts/shap_proxy_rank.py (DEFERRED — G4, G5)
```

### Stage 6 — Challenger Training

```
Action: Train LightGBM on same X_train / X_val / X_test split as champion.
        Apply same Platt calibration.
        Run same evaluation suite.
        Enter shadow scoring mode — challenger runs on all applications
        alongside champion but its score is NEVER used for real decisions.

Output: outputs/evidence/challenger_promotion_decision.json
        outputs/models/challenger_lgbm.pkl
Source: scripts/train_challenger.py, scripts/champion_challenger_compare.py (DEFERRED — G6)
```

### Stage 7 — Champion/Challenger 5-Gate Evaluation

```
Action: Head-to-head comparison at 5 gates (challenger must pass ALL to be promoted):
  Gate 1: PR AUC delta ≥ 0.001 (challenger must exceed champion by meaningful margin)
  Gate 2: ROC AUC delta ≥ 0.001
  Gate 3: ECE must not regress (challenger ECE ≤ champion ECE × 1.10)
  Gate 4: Calibration curve must not diverge at threshold points (0.06, 0.28)
  Gate 5: DeLong AUC test p < 0.05 (difference is statistically significant)

Gate outcomes:
  ALL PASS → Challenger promoted; champion archived; policy version log updated
  ANY FAIL → Champion retained; challenger archived with documented failure reason

Expected outcome (from source-reference SR-10, SR-11):
  Gate 1: FAIL — delta ~0.0002, below 0.001 threshold
  Champion retained. This is the documented outcome — PulseGuard will verify at G6.

Output: outputs/evidence/challenger_promotion_decision.json
Source: scripts/champion_challenger_compare.py (DEFERRED — G6)
```

## 4. PHASE 3 — DECISION PIPELINE (PER-APPLICATION)

**Execution context:** Application-level. In the production simulation, runs on the synthetic application batch. In real deployment (not this portfolio), would run per API call.

> **SIMULATED:** All applications in PulseGuard are synthetic or Home Credit records.
> No real loan decisions are made. This pipeline is a production-pattern simulation.

### Stage 8 — Policy Hard Gate

```
Action: Apply 6 deterministic hard rules BEFORE any model call.
        FOIR is always RECOMPUTED from raw inputs — never trusted from the application.

FOIR formula:
  FOIR = (EXISTING_OBLIGATIONS_MONTHLY + AMT_ANNUITY) / AMT_INCOME_TOTAL

  ⚠️ SYNTHETIC FIELD:
    EXISTING_OBLIGATIONS_MONTHLY is not present in Home Credit.
    Home Credit provides: AMT_INCOME_TOTAL (income), AMT_ANNUITY (proposed EMI).
    Existing monthly obligations are SYNTHETIC, generated as:
      EXISTING_OBLIGATIONS_MONTHLY ~ Uniform(0.05, 0.25) × AMT_INCOME_TOTAL
    This synthetic generation is:
      (a) Documented in every artifact referencing FOIR
      (b) Labeled SYNTHETIC in the policy output schema
      (c) Not claimed as real borrower data

Hard rules (in evaluation order):
  Rule 1 — FOIR:      FOIR > 0.65 → HARD_REJECT (income obligation too high)
  Rule 2 — LTV:       AMT_CREDIT / AMT_GOODS_PRICE > 0.90 → HARD_REJECT (loan-to-value exceeded)
  Rule 3 — Income:    AMT_INCOME_TOTAL < 15000 → HARD_REJECT (below income floor)
  Rule 4 — Age:       DAYS_BIRTH > −6570 (< 18 years) → HARD_REJECT (below minimum age)
  Rule 5 — Age:       DAYS_BIRTH < −25550 (> 70 years) → HARD_REJECT (above maximum age)
  Rule 6 — Region:    REGION_RATING_CLIENT == 3 → REFER (high-risk region; escalate to REVIEW)

Gate outcomes:
  HARD_REJECT → Write audit trail JSONL entry; skip model scoring; return REJECT
  REFER       → Route to REVIEW queue; continue with model scoring
  PASS        → Continue to model scoring

Output: Policy hard gate fields in audit_record.jsonl (schema in G2_data_contracts.md)
Source: src/foir_engine.py, src/policy_gate.py (DEFERRED — G7)
```

### Stage 9 — Champion Score

```
Action: Load champion XGBoost (calibrated). Score the application.
        Output is a calibrated probability: PD ∈ [0, 1].

Output field: pd_score_champion (float)
Source: scripts/train_champion.py output model, loaded at decision time (DEFERRED — G7)
```

### Stage 10 — Challenger Shadow Score

```
Action: Load challenger LightGBM (calibrated). Score the SAME application.
        Output is logged only — NOT used for the actual decision.
        Used for ongoing head-to-head comparison at next gate cycle.

Output field: pd_score_challenger (float)
              challenger_decision (APPROVE/REVIEW/REJECT based on challenger thresholds)
              challenger_agrees_with_champion (boolean)
Source: scripts/train_challenger.py output model (DEFERRED — G6/G7)
```

### Stage 11 — Decision Router

```
Action: Apply policy threshold to champion PD score.

Decision logic:
  pd_score_champion < 0.06              → APPROVE
  0.06 ≤ pd_score_champion < 0.28       → REVIEW (routed to human underwriter queue)
  pd_score_champion ≥ 0.28              → REJECT

Policy version: v1.0 (initial)
Policy change event: v1.1 at Day 12 of lifecycle simulation
  v1.1 change: lower APPROVE threshold from 0.06 → 0.05 (tightening)
  Effect: some previously-approved applicants move to REVIEW band
  Attribution: approval rate decomposition must isolate this effect from model drift

Output field: decision (enum: APPROVE / REVIEW / REJECT)
              policy_version (str: "v1.0" or "v1.1")
Source: src/decision_router.py (DEFERRED — G7)
```

### Stage 12 — SHAP Adverse Action Factors

```
Action: For applications with decision = REJECT or REVIEW:
        Compute SHAP values using shap.Explainer (champion model, background sample).
        Identify top-3 features by |SHAP| value for the individual applicant.
        Map these to adverse action reason codes.
        These are illustrative ECOA-pattern codes, not filed adverse action notices.

Output fields: adverse_action_code_1, adverse_action_code_2, adverse_action_code_3
               adverse_action_shap_1, adverse_action_shap_2, adverse_action_shap_3
Source: src/adverse_action_codes.py (DEFERRED — G7)

Note: Home Credit features mapped to reason code categories:
  EXT_SOURCE_2/3 → "Insufficient external credit bureau score"
  AMT_INCOME_TOTAL → "Insufficient income"
  DAYS_EMPLOYED → "Insufficient employment history"
  AMT_CREDIT → "Requested credit amount too high"
  FOIR (synthetic) → "Debt-to-income ratio exceeds threshold"
```

### Stage 13 — Audit Trail Entry

```
Action: Write one JSONL record per application decision.
        Record is tamper-evident (hash of inputs + outputs concatenated).
        Written to outputs/evidence/audit_trail.jsonl.

See G2_data_contracts.md § audit_record for full schema.
Source: src/audit_logger.py (DEFERRED — G7)
```

## 5. PHASE 4 — MONITORING PIPELINE

**Execution context:** Batch, daily over the 30-day simulated lifecycle.

> **SIMULATED:** All batches are generated by scripts/seed_lifecycle.py.
> Drift events are injected synthetically. This is explicitly labeled in all artifacts.

### Stage 14 — Drift Monitor

```
Action: For each daily batch (B00–B29):
        Compute PSI (Population Stability Index) per feature vs. training reference.
        Compute KS statistic per feature vs. training reference.

PSI thresholds:
  PSI < 0.10  → STABLE (no concern)
  0.10 ≤ PSI < 0.20 → WARN (monitor closely; log alert)
  PSI ≥ 0.20  → ALERT (trigger model review; log retraining consideration)

Retraining trigger:
  PSI ALERT on any key feature (EXT_SOURCE_2, EXT_SOURCE_3, AMT_CREDIT) for
  ≥ 2 consecutive batches → formal retraining trigger documented in model card.

Lifecycle events (SIMULATED):
  Day 0:  Reference batch (training distribution)
  Day 7:  EXT_SOURCE_2 distribution shift begins; PSI crosses 0.10 → WARN
  Day 14: EXT_SOURCE_2 shift magnitude −0.18; PSI ~ 0.23 → ALERT

Also computes:
  Approval rate per batch (used by Stage 16)
  Review rate per batch
  Reject rate per batch

Output: outputs/evidence/drift_report.json (schema in G2_data_contracts.md)
        outputs/plots/psi_time_series.png
Source: scripts/drift_monitor.py, scripts/seed_lifecycle.py (DEFERRED — G4)
```

### Stage 15 — Fairness Audit

```
Action: Compute fairness metrics on full scored dataset (test set or production batch).

Metrics:
  Disparate Impact (DI): approval_rate(group=F) / approval_rate(group=M)
    Target band: 0.80–1.25 (80% rule)
    ≥ 0.80 → PASS
    < 0.80 → FAIL (potential adverse impact on female applicants)

  Equal Opportunity (EOpp) gap: TPR(group=F) − TPR(group=M)
    Target: < 5pp gap
    < 5pp → PASS
    ≥ 5pp → WARN; ≥ 10pp → FAIL

  SHAP rank check:
    CODE_GENDER_F mean |SHAP| rank must not be in top 5 features by importance.
    (Protected attribute proxy check)

Scope: CODE_GENDER is the only group variable analyzed (from Home Credit).
       Not a real protected class in a deployed system — methodology is correct,
       results are dataset-specific.

Output: outputs/evidence/fairness_report.json (schema in G2_data_contracts.md)
        outputs/plots/shap_beeswarm.png
Source: scripts/fairness_audit.py, scripts/shap_proxy_rank.py (DEFERRED — G5)
```

### Stage 16 — Approval Rate Decomposition

```
Action: Attribute approval rate change across batches to two competing causes:
  (a) Model drift — PD scores shifting due to distribution change
  (b) Policy version change — threshold change at Day 12 (v1.0 → v1.1)

Method:
  For each batch, compute:
    counterfactual_approval_rate = approval rate if v1.0 thresholds applied
    actual_approval_rate = approval rate at current policy version

  Difference = policy_effect
  (actual_approval_rate − counterfactual_approval_rate) decomposed as:
    model_effect = PD score distribution shift
    policy_effect = threshold change contribution

Output: outputs/evidence/decision_simulation_report.json (§ approval_rate_decomposition)
Source: scripts/approval_rate_decomposition.py (DEFERRED — G7)

Interview framing:
  "If approval rate drops from 72% to 65%, I can attribute how much came from
  the model seeing a riskier population versus how much came from the threshold
  tightening at policy v1.1. Without this separation, you can't tell which lever
  to pull."
```

  - Summary of all gate results
